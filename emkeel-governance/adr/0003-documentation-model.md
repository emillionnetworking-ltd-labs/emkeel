# 3. Emkeel documentation model

- Status: accepted
- Date: 2026-06-16
- Deciders: operator (EMillion Networking)

## Context

Document handling depended on the prompt (ephemeral), not on a durable rule: a doc landed in the
right place only because someone said so in a prompt. In a real governed repo (em-ecosystem) this
let framework-era docs rot — orphaned snapshots in `docs/` that drifted from the code and
masqueraded as current (ECO-13). And there are several distinct kinds of document, each generated by
a different path; a single ad-hoc "docs rule" is too thin. The model has to be named so the rule
derives from it, not from memory.

## Decision

Emkeel recognizes **seven document types**, each with a purpose, a generation path, a home, and
whether it is gated:

| Type | Purpose | Generated by | Home | Gated |
|------|---------|--------------|------|-------|
| strategy | living north star (where + why) | `/strategy` skill | `emkeel-governance/strategy/` | yes |
| ADR | a discrete decision (what + why) | authored when a decision crystallizes | `emkeel-governance/adr/` | partial |
| spec | a feature's acceptance criteria | per `feat/` ticket (the loop) | `emkeel-governance/specs/` | yes |
| record | audit / what happened | an audit task | `emkeel-governance/records/` | no |
| product reference | what the system IS (architecture, how-to) | authored deliberately, status-headed | `docs/` | no |
| historical reference | why we got here (preserved) | deliberate rescue | `docs/archive/` | no |
| generated | technical artifact (OpenAPI, typedoc) | CI from the code | regenerated, not hand-stored | no |

Codify the `docs/` half (the part that had no prior path) into the **scaffolded `AGENTS.md`**
(`src/emkeel/init.py`): `docs/` = product reference (living); governance never lives in `docs/`;
code-mirroring specs are regenerated, not snapshotted; `docs/archive/` holds historical docs with a
status header; a dead doc is deleted (git is the archive), not parked.

## Consequences

- The convention is **durable** (a written contract), not prompt-dependent — any agent in any
  governed repo reads it. The drift that orphaned em-ecosystem's `docs/` won't recur silently.
- `docs/` stays a top-level product folder, **outside** `emkeel-governance/` (which is
  `export-ignore` + gated). Reference docs are visible/shippable and aren't gate-able.
- **No hard gate** on `docs/` — reference accuracy isn't a computable fact. The discipline is the
  convention + status headers, not enforcement.
- A `docs/README` scaffold is deferred (optional) — the `AGENTS.md` convention is the core.

## Alternatives considered

- Put `docs/` inside `emkeel-governance/` → rejected: that folder is `export-ignore` (docs should be
  visible) and gated (reference docs can't be gated); it also conflates governance with description.
- A hard gate on docs (staleness / structure) → rejected: "is this doc current?" is not computable.
- Leave it prompt-dependent → rejected: that is exactly what orphaned em-ecosystem's `docs/`.
